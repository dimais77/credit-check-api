# Credit Check API

Сервис предварительной проверки пакета документов по льготному кредиту. Принимает название программы (`federal` / `regional`) и загруженные файлы, определяет тип каждого документа по имени, проверяет комплектность пакета, формат и размер файлов, выставляет статус и сохраняет результат в PostgreSQL.

Стек: Python 3.11, FastAPI, SQLAlchemy 2.0 (async), PostgreSQL, Alembic, Docker Compose.

## Запуск

Нужен Docker с Compose v2. Весь стек поднимается одной командой:

```bash
cp .env.example .env
docker compose up --build
```

Поднимаются три сервиса: `db` (PostgreSQL), одноразовый `migrate` (накатывает миграции и выходит) и `app` - API на http://localhost:8000, который стартует только после успешных миграций. Swagger - на `/docs`, плюс есть `/health` (liveness) и `/ready` (readiness с пингом БД).

Пример запроса:

```bash
curl -X POST http://localhost:8000/api/checks \
  -F "program=federal" \
  -F "files=@договор_47.pdf" \
  -F "files=@спецификация.pdf" \
  -F "files=@счёт.pdf" \
  -F "files=@акт.pdf"
```

## Тесты

Тестам нужна отдельная БД `credit_test` на том же PostgreSQL - поднимите `db` и создайте её один раз:

```bash
docker compose up -d db
docker compose exec -T db psql -U credit -d credit -c "CREATE DATABASE credit_test;"

uv sync
cp src/.env.example src/.env
uv run pytest
```

Тесты гоняются во вложенной транзакции с откатом, так что в `credit_test` ничего не оседает, а файлы пишутся во временный каталог.

## Технологии

FastAPI и Pydantic v2 закрывают async HTTP, валидацию на границе, multipart и авто-`/docs`. Данные - в PostgreSQL через async SQLAlchemy 2.0 поверх asyncpg; типы программы, статуса и документа хранятся как native enum. Миграции - Alembic, отдельным шагом в compose.

Зависимости и сборка - на `uv` (быстрый резолвер, `uv.lock`, multi-stage Docker). Без CI: ruff, mypy --strict, pytest запускаются локально.

## Архитектура

Код в src/. Бизнес-правила - в функциях без БД и файлов, ввод-вывод собран отдельно.

```
src/
├── main.py           # сборка приложения, роутеры, обработчики ошибок
├── api/routes/       # эндпоинты: checks, health
├── schemas/          # Pydantic-модели запроса/ответа, Page[T]
├── services/         # доменное ядро + оркестрация проверки
│   ├── document_classifier.py   # тип документа по имени файла
│   ├── validation.py            # формат/размер/имя + комплектность
│   ├── status.py                # итоговый статус и причина
│   └── check_service.py         # run_check: весь сценарий проверки
├── repositories/     # доступ к БД + входные DTO
├── models/           # ORM: Check, Document, Issue
├── storage/          # запись файлов в хранилище
├── core/             # config, database, logging, exceptions
└── migrations/       # Alembic
```

`document_classifier`, `validation` и `status` - чистые функции без БД, ФС и настроек (конфиг приходит параметрами), под ними обычные unit-тесты. Весь ввод-вывод собран в `check_service.run_check` и репозиториях.

Сценарий `POST /api/checks`: по каждому файлу определяется тип по имени, проверяются формат/размер/распознавание, байты пишутся в хранилище; дальше - комплектность пакета под выбранную программу, расчёт статуса и запись `Check` с документами и issues одной транзакцией. Файл сохраняется до коммита: если коммит упадёт, на диске останется осиротевший файл (он логируется), а не строка в БД без файла.

Документы в ответе отсортированы по порядку документооборота: договор → спецификация → счёт → акт, нераспознанные в конце. Акт/УПД - закрывающий документ, поэтому идёт после счёта. Порядок хранится в БД, так что POST и GET отдают его одинаково.

Под целевое видение из БТ заложены точки роста: хранилище можно заменить на S3, поле `extracted` зарезервировано под извлечение реквизитов, записи `Check` - под версионность.

## Переменные окружения

Конфиг разнесён на два файла. Корневой `.env` - для Compose, `src/.env` - для приложения (Pydantic Settings, префикс `APP_CONFIG__`, вложенность через `__`). В контейнере приложение получает `src/.env`, а URL базы переопределяется на хост `db`.

Compose-переменные в корневом `.env.example`: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `POSTGRES_PORT` (порт БД на хосте, слушает `127.0.0.1`) и `APP_PORT`.

Приложение (`src/.env.example`) - главное:

| Переменная | Назначение |
|---|---|
| `APP_CONFIG__DB__URL` | DSN основной БД (asyncpg) |
| `APP_CONFIG__DB__TEST_URL` | DSN тестовой БД для pytest |
| `APP_CONFIG__STORAGE__DIR` | каталог файлового хранилища |
| `APP_CONFIG__STORAGE__MAX_FILE_SIZE_MB` | лимит размера файла, МБ |

Остальные (`LOG_LEVEL`, `DB__ECHO`, размеры пула) идут со значениями по умолчанию - трогать не обязательно.

## Решения и трактовки ТЗ

- **Статусы:** `error` (нет обязательного документа) → `rejected`; `warning` (не распознан формат/имя, размер больше лимита) без ошибок → `check_in_progress`; иначе `approved`. Принцип из БТ: при неопределённости не выдаём `approved`, а отправляем пакет на ручную проверку.
- **HTTP-коды.** 4xx - только про некорректный запрос: нет файлов → 400, неверная `program` → 422, не найдено → 404. А нарушения по документам - это `issues` в успешном 201, не ошибка HTTP.
- `status` возвращаю как `rejected`, а не `reject` из примера ТЗ - считаю это опечаткой, ориентируюсь на нормативный пункт 6.
- `extracted` пока `null`: извлечение реквизитов (OCR/LLM) вне рамок MVP, поле оставлено на будущее.

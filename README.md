# Credit Check API

Сервис предварительной проверки пакета документов по льготному кредиту. Принимает название программы (`federal` / `regional`) и загруженные файлы, определяет тип каждого документа по имени, проверяет комплектность пакета, формат и размер файлов, выставляет статус и сохраняет результат в PostgreSQL.

Стек: Python 3.11, FastAPI, SQLAlchemy 2.0 (async), PostgreSQL, Alembic, Docker Compose.

## Запуск

Нужен Docker с Compose v2. Весь стек поднимается одной командой (без предварительной настройки):

```bash
docker compose up
```

Compose-переменные (`POSTGRES_*`, `APP_PORT`) имеют дефолты непосредственно в `docker-compose.yml`, приложение получает DSN базы данных через `environment`, поэтому `.env` и `src/.env` для запуска не нужны. Для переопределения дефолтов необходимо создать корневой `.env` (см. пример `.env.example`).

Поднимаются три сервиса: 
- `db` (PostgreSQL),
- `migrate` (одноразовый, делает миграции и выходит)
- `app` - API на http://localhost:8000 - стартует после успешных миграций. 
Swagger - на `/docs`, есть `/health` (liveness) и `/ready` (readiness с проверкой БД).

Пример запроса (`Idempotency-Key` сейчас опционален - но он защищает от дублей при повторной отправке, см. раздел "Решения и трактовки ТЗ"):

```bash
curl -X POST http://localhost:8000/api/checks \
  -H "Idempotency-Key: $(uuidgen)" \
  -F "program=federal" \
  -F "files=@договор_47.pdf" \
  -F "files=@спецификация.pdf" \
  -F "files=@счет.pdf" \
  -F "files=@акт.pdf"
```

## Тесты

Для Тестов нужна тестовая БД `credit_test` (на том же PostgreSQL) - поднимите `db` и создайте ее один раз:

```bash
docker compose up -d db
docker compose exec -T db psql -U credit -d credit -c "CREATE DATABASE credit_test;"

uv sync

uv run pytest
```

## Технологии

FastAPI и Pydantic v2 - закрывают async HTTP, валидацию, multipart и авто-`/docs`. 
Данные - в PostgreSQL через async SQLAlchemy 2.0 поверх asyncpg; типы программы, статуса и документа хранятся как native enum.
Миграции - Alembic, отдельным шагом в compose.
Зависимости и сборка - на `uv` (быстрый резолв, `uv.lock`, multi-stage Docker). 
Без CI. ruff, mypy, pytest запускаются локально.

## Архитектура

Код в src/. Бизнес-правила - в функциях (без БД и файлов), ввод-вывод собран отдельно.

```
src/
├── main.py           # сборка приложения, роутеры, обработчики ошибок
├── api/routes/       # эндпоинты: checks, health
├── schemas/          # Pydantic-модели запроса/ответа, CursorPage[T]
├── services/         # доменное ядро + оркестрация проверки
│   ├── document_types.py        # единый реестр метаданных типов (подпись, паттерны)
│   ├── document_classifier.py   # тип документа по имени файла
│   ├── validation.py            # формат/размер/имя + комплектность
│   ├── status.py                # итоговый статус и причина
│   └── check_service.py         # run_check: весь сценарий проверки
├── repositories/     # доступ к БД + DTO
├── models/           # ORM: Check, Document, Issue
├── storage/          # запись файлов в хранилище
├── core/             # config, database, logging, exceptions
└── migrations/       # Alembic
```

`document_classifier`, `validation` и `status` - чистые функции без БД, ФС и настроек (конфиг - параметрами), под ними обычные unit-тесты. Весь ввод-вывод собран в `check_service.run_check` и репозиториях.

Сценарий `POST /api/checks`: 
по каждому файлу определяется тип по имени, байты стримятся в хранилище чанками с хешированием, чтение обрывается по лимиту размера файла; 
дальше - комплектность пакета под выбранную программу, расчет статуса и запись `Check` с документами и issues одной транзакцией. 
Файлы сохраняются до коммита, поэтому осиротевших строк в БД нет. 
При любом сбое IO-сценария (запись файла или коммит) весь каталог `{check_id}/` удаляется из хранилища, событие логируется, файлов не остается.

Файл сверх лимита (`MAX_FILE_SIZE_MB`) не сохраняется: стрим обрывается, `storage_path` документа остается пустым, а сам пакет получает `warning` с причиной и уходит на ручную проверку (`check_in_progress`) - это защищает хранилище и память от слишком больших файлов, не отклоняя запрос как ошибочный.

Документы в ответе отсортированы по порядку документооборота: 
* договор -> спецификация -> счет -> акт, нераспознанные в конце. 
* Акт/упд - закрывающий документ, идет после счета. 
* Порядок хранится в БД, так что POST и GET отдают его одинаково.

Под целевое видение из БТ: хранилище можно заменить на S3, поле `extracted` зарезервировано под извлечение реквизитов, записи `Check` - под версионность.

Лимит размера задан пофайлово (`MAX_FILE_SIZE_MB` - требование ТЗ). Ограничение совокупного размера запроса ТЗ не задает и сейчас оставлено вне рамок.

## Переменные окружения

Конфиг разнесен на два файла. Корневой `.env` - для Compose, `src/.env` - для приложения (Pydantic Settings, префикс `APP_CONFIG__`, вложенность через `__`). 
В Compose они опциональны: Compose-переменные имеют дефолты, а приложение в контейнере получает DSN базы и каталог хранилища напрямую через `environment` (хост `db`), остальное - из дефолтов конфига. `src/.env` нужен при локальном запуске приложения и тестов вне контейнера.

Compose-переменные в корневом `.env.example`: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `POSTGRES_PORT` (порт БД на хосте, слушает `127.0.0.1`) и `APP_PORT`.

Приложение (`src/.env.example`):

| Переменная | Назначение                  |
|---|-----------------------------|
| `APP_CONFIG__DB__URL` | DSN основной БД (asyncpg)   |
| `APP_CONFIG__DB__TEST_URL` | DSN тестовой БД для pytest  |
| `APP_CONFIG__STORAGE__DIR` | каталог файлового хранилища |
| `APP_CONFIG__STORAGE__MAX_FILE_SIZE_MB` | лимит размера файла         |

Остальные (`LOG_LEVEL`, `DB__ECHO`, размеры пула, таймауты БД `DB__STATEMENT_TIMEOUT_MS` / `DB__LOCK_TIMEOUT_MS`) указаны со значениями по умолчанию.

## Решения и трактовки ТЗ

**Статусы:** 
- `error` (нет обязательного документа) -> `rejected`; 
- `warning` (не распознан формат/имя, размер больше лимита) без ошибок -> `check_in_progress`; 
- иначе `approved`. Принцип из БТ: при неопределенности не выдаем `approved`, а отправляем пакет на ручную проверку.

**Тип документа по имени.** 
ТЗ требует анализ "по паттерну имени": тип определяется по ключевым словам на границах слов (`\b`) с учетом словоформ, а не по подстроке. При неоднозначности приоритет "специфичное -> общее": `договор` проверяется последним, т.к. входит в имена смежных документов, поэтому `спецификация_к_договору` -> `specification`, `акт_к_договору` -> `act`.

**HTTP-коды.** 
- 4xx - только про некорректный запрос: нет файлов -> 400, неверная `program` -> 422, не найдено -> 404. 
- Нарушения по документам - это `issues` в успешном 201, не ошибка HTTP.
- `status` возвращаю как `rejected` (не `reject` - видимо опечатка в примере из ТЗ) - ориентируюсь на нормативный пункт 6.
- **`program` в ответе `POST/GET /api/checks/{id}`.** В примере ответа ТЗ его нет, но список (`GET /api/checks`) программу отдает - поэтому карточка тоже возвращает `program`.
- **«Анализ по количеству» (п.4 ТЗ)** понимаю как `documents_count` в списке + наличие каждого обязательного типа. Множественность (несколько счетов/актов в пакете) допустима и на статус не влияет: комплектность проверяется по присутствию типа, поэтому несколько документов одного типа не отклоняют пакет.
- `extracted` пока `null`: извлечение реквизитов (OCR/LLM) вне рамок ТЗ, поле оставил на будущее.

**История.** «История» из ТЗ (пп. 2-3) - это журнал проверок: список всех и карточка каждой. Версионность документов и разграничение прав - из видения БТ, вне пунктов 1-9.

**`Idempotency-Key` - опциональный заголовок `POST /api/checks`** (защита от дублей при ретраях, повторных отправок; сделал необязательным, чтобы не ломать контракт ТЗ для клиента, который его не шлет). 
Если передан: 
- клиентский UUID, повтор с тем же ключом и тем же пакетом файлов возвращает исходный `Check` без повторной обработки; 
- тот же ключ с другим пакетом -> 422; 
- параллельный запрос с тем же ключом, пока первый еще обрабатывается -> 409. 
- Без заголовка защиты от дублей нет - каждый POST создает новый `Check`.

**Пагинация `GET /api/checks`**   
- метод использует курсорную пагинацию по составному ключу (checked_at, id). Это исключает дублирование записей в растущем журнале.

# Credit Check API

Сервис предварительной проверки пакета документов по льготному кредиту. Принимает название программы (`federal` / `regional`) и загруженные файлы, определяет тип каждого документа по имени, проверяет комплектность пакета, формат и размер файлов, выставляет статус и сохраняет результат в PostgreSQL.

Стек: Python 3.11, FastAPI, SQLAlchemy 2.0 (async), PostgreSQL, Alembic, Docker Compose.

## Запуск

Нужен Docker с Compose v2. Весь стек поднимается одной командой:

```bash
cp .env.example .env
docker compose up --build
```

Поднимаются три сервиса: `db` (PostgreSQL), одноразовый `migrate` (накатывает миграции и выходит) и `app` - API на http://localhost:8000, который стартует только после успешных миграций. Swagger - на `/docs`, плюс есть `/health` (liveness) и `/ready` (readiness с пингом БД).

Пример запроса (`Idempotency-Key` опционально - защищает от дублей при повторной отправке, см. "Решения и трактовки ТЗ"):

```bash
curl -X POST http://localhost:8000/api/checks \
  -H "Idempotency-Key: $(uuidgen)" \
  -F "program=federal" \
  -F "files=@договор_47.pdf" \
  -F "files=@спецификация.pdf" \
  -F "files=@счет.pdf" \
  -F "files=@акт.pdf"
```

## Тесты

Тестам нужна отдельная БД `credit_test` на том же PostgreSQL - поднимите `db` и создайте ее один раз:

```bash
docker compose up -d db
docker compose exec -T db psql -U credit -d credit -c "CREATE DATABASE credit_test;"

uv sync
cp src/.env.example src/.env
uv run pytest
```

Тесты прогоняются во вложенной транзакции с откатом, в `credit_test` ничего не оседает, файлы пишутся во временный каталог.

## Технологии

FastAPI и Pydantic v2 закрывают async HTTP, валидацию на границе, multipart и авто-`/docs`. 
Данные - в PostgreSQL через async SQLAlchemy 2.0 поверх asyncpg; типы программы, статуса и документа хранятся как native enum.
Миграции - Alembic, отдельным шагом в compose.
Зависимости и сборка - на `uv` (быстрый резолв, `uv.lock`, multi-stage Docker). 
Без CI: ruff, mypy --strict, pytest запускаются локально.

## Архитектура

Код в src/. Бизнес-правила - в функциях без БД и файлов, ввод-вывод собран отдельно.

```
src/
├── main.py           # сборка приложения, роутеры, обработчики ошибок
├── api/routes/       # эндпоинты: checks, health
├── schemas/          # Pydantic-модели запроса/ответа, Page[T]
├── services/         # доменное ядро + оркестрация проверки
│   ├── document_types.py        # единый реестр метаданных типов (подпись, паттерны)
│   ├── document_classifier.py   # тип документа по имени файла
│   ├── validation.py            # формат/размер/имя + комплектность
│   ├── status.py                # итоговый статус и причина
│   └── check_service.py         # run_check: весь сценарий проверки
├── repositories/     # доступ к БД + DTO
├── models/           # ORM: Check, Document, Issue
├── storage/          # запись файлов в хранилище
├── core/             # config, database, logging, exceptions
└── migrations/       # Alembic
```

`document_classifier`, `validation` и `status` - чистые функции без БД, ФС и настроек (конфиг - параметрами), под ними обычные unit-тесты. Весь ввод-вывод собран в `check_service.run_check` и репозиториях.

Сценарий `POST /api/checks`: по каждому файлу определяется тип по имени, проверяются формат/размер/распознавание, байты пишутся в хранилище; дальше - комплектность пакета под выбранную программу, расчет статуса и запись `Check` с документами и issues одной транзакцией. Файлы сохраняются до коммита, поэтому строки в БД без файла быть не может. Обратный случай контролируется: при любом сбое IO-сценария (запись файла или коммит) весь каталог чека `{check_id}/` удаляется из хранилища, событие логируется - осиротевших файлов не остается.

Документы в ответе отсортированы по порядку документооборота: договор -> спецификация -> счет -> акт, нераспознанные в конце. Акт/упд - закрывающий документ, идет после счета. Порядок хранится в БД, так что POST и GET отдают его одинаково.

Под целевое видение из БТ заложены точки роста: хранилище можно заменить на S3, поле `extracted` зарезервировано под извлечение реквизитов, записи `Check` - под версионность.

## Переменные окружения

Конфиг разнесен на два файла. Корневой `.env` - для Compose, `src/.env` - для приложения (Pydantic Settings, префикс `APP_CONFIG__`, вложенность через `__`). В контейнере приложение получает `src/.env`, а URL базы переопределяется на хост `db`.

Compose-переменные в корневом `.env.example`: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `POSTGRES_PORT` (порт БД на хосте, слушает `127.0.0.1`) и `APP_PORT`.

Приложение (`src/.env.example`) - главное:

| Переменная | Назначение |
|---|---|
| `APP_CONFIG__DB__URL` | DSN основной БД (asyncpg) |
| `APP_CONFIG__DB__TEST_URL` | DSN тестовой БД для pytest |
| `APP_CONFIG__STORAGE__DIR` | каталог файлового хранилища |
| `APP_CONFIG__STORAGE__MAX_FILE_SIZE_MB` | лимит размера файла, МБ |

Остальные (`LOG_LEVEL`, `DB__ECHO`, размеры пула, таймауты БД `DB__STATEMENT_TIMEOUT_MS` / `DB__LOCK_TIMEOUT_MS`) идут со значениями по умолчанию - трогать не обязательно.

## Решения и трактовки ТЗ

- **Статусы:** `error` (нет обязательного документа) -> `rejected`; `warning` (не распознан формат/имя, размер больше лимита) без ошибок -> `check_in_progress`; иначе `approved`. Принцип из БТ: при неопределенности не выдаем `approved`, а отправляем пакет на ручную проверку.
- **HTTP-коды.** 4xx - только про некорректный запрос: нет файлов -> 400, неверная `program` -> 422, не найдено -> 404. А нарушения по документам - это `issues` в успешном 201, не ошибка HTTP.
- `status` возвращаю как `rejected` (а не `reject` из примера в ТЗ) - ориентируюсь на нормативный пункт 6.
- `extracted` пока `null`: извлечение реквизитов (OCR/LLM) вне рамок ТЗ, поле оставлено на будущее.
- **`Idempotency-Key` - опциональный заголовок `POST /api/checks`** (защита от дублей при ретраях по таймауту; не обязателен, чтобы не ломать контракт ТЗ для клиента, который его не шлет). Если передан: клиентский UUID, повтор с тем же ключом и тем же пакетом файлов возвращает исходный `Check` без повторной обработки; тот же ключ с другим пакетом -> 422; параллельный запрос с тем же ключом, пока первый еще обрабатывается -> 409. Без заголовка защиты от дублей нет - каждый POST создает новый `Check`.
